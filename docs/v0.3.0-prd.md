# PRD: MeetSum v0.3.0 — Operational Maturity

## Overview

MeetSum v0.2.0 established the intelligence foundation: share lifecycle, room suggestions, summary templates, audit logging, and rate limiting. However, most of these capabilities exist as **backend-only primitives** — the wiring to actual middleware, UI surfaces, and operational visibility is incomplete.

v0.3.0 closes the loop. Every capability added in v0.1.0–v0.2.0 must become **self-serve**: an admin should be able to manage API keys, see system health, and configure rooms from the UI without SSH access. An operator should be able to run Hebrew ASR evaluations, apply summary templates, and trigger backup/restore from documented commands. The public share page should feel polished enough to send to a client.

**Who is this for?** Solo operators and small teams running MeetSum on a self-hosted VPS to capture, process, and share Google Meet intelligence.

**What problem does it solve?** v0.2.0 built the engine; v0.3.0 puts the steering wheel and dashboard in the driver's seat.

## User Stories

- As an **admin**, I want to create, revoke, and list API keys from the UI so that I don't need SSH access to manage automation credentials.
- As an **admin**, I want to see system health metrics (queue depth, failed jobs, storage growth, sync lag) so that I can diagnose issues without Docker exec.
- As an **operator**, I want rate limiting to automatically protect public endpoints so that a shared meeting link can't be used to DoS the instance.
- As an **operator**, I want summary templates wired into the AI pipeline so that a "Sales Call" meeting gets sales-specific intelligence, not generic output.
- As a **user**, I want a Room detail page so that I can see all meetings, tasks, and participants in a project context.
- As a **user**, I want speaker-to-person assignment so that "Speaker 1" in the transcript becomes "Alice" everywhere.
- As a **user**, I want to export meetings as DOCX so that I can share them with people who don't use Markdown.
- As an **operator**, I want MinIO + Postgres backup to run as a single documented command so that I can restore the system after a failure.
- As an **operator**, I want to evaluate Hebrew ASR quality against private reference transcripts so that I can measure whether model upgrades improve accuracy.

## Functional Requirements

### Epic 1: Admin Self-Service (P0 — High Value)

#### Feature 1.1: API Key Management UI
- Admin page at `/settings/api-keys` showing active API keys (masked)
- Create new key with optional label and expiration
- Revoke key with confirmation dialog
- Copy-to-clipboard on creation (key shown only once)
- Audit log entry on create/revoke

#### Feature 1.2: Rate Limit Middleware Integration
- Wire `checkRateLimit()` into Next.js middleware for:
  - Public share pages (`/share/:token`) — 60 req/min per token
  - AI processing endpoints — 10 req/min per IP
  - Export endpoints — 20 req/min per IP
  - Ask/memory endpoints — 30 req/min per IP
- Return `429 Too Many Requests` with `Retry-After` header
- Apply `X-RateLimit-*` headers to all API responses

#### Feature 1.3: Operational Dashboard
- New admin page at `/settings/operations` showing:
  - Queue depth (pending/active/failed jobs)
  - Google Calendar sync freshness
  - Google Meet artifact sync freshness
  - Storage usage (MinIO + Postgres)
  - Recent audit log entries
  - AI run stats (latency p50/p95, provider distribution)
- Auto-refresh every 30 seconds

### Epic 2: Intelligence Pipeline Completion (P0 — High Value)

#### Feature 2.1: Summary Template → AI Pipeline Wiring
- `getSummaryTemplatePrompt(template)` addendum appended to summarization prompt
- Template selection from app settings (`summaryTemplate` field already exists)
- Per-meeting override via meeting detail UI dropdown
- Template name included in AI run metadata

#### Feature 2.2: Speaker-to-Person Assignment UI
- Modal in Transcript tab showing speaker labels and participant names
- Drag-and-drop or dropdown assignment
- Persist mapping via `POST /api/meetings/:id/speakers/assign`
- Retroactively update all transcript segments with the assigned name
- Propagate speaker names to summary citations and action item owners

#### Feature 2.3: Hebrew ASR Evaluation Script
- `npm run asr:evaluate` runs against private manifest at `.secrets/asr-eval/manifest.json`
- Evaluates WER per sample, per provider, per language
- Outputs JSON report with: sampleId, provider, model, WER, latencyMs
- Docs already exist at `docs/asr-evaluation.md`; wire the actual script

### Epic 3: Room & Memory UX (P1 — High Value)

#### Feature 3.1: Room Detail Page
- Page at `/rooms/:id` showing:
  - Room header (name, description, color, meeting count)
  - Linked meetings list (card-style, sortable by date)
  - Open tasks across all meetings
  - Recurring participants with meeting frequency

#### Feature 3.2: Room Suggestions UI
- Suggestion banner on meeting detail when the meeting has no room context
- Banner shows suggested room name + reason from `GET /api/rooms/suggestions`
- One-click "Create Room" or "Add to Existing Room"
- Suggestions also shown on the Rooms list page sidebar

### Epic 4: Export & Share Polish (P1 — Medium Value)

#### Feature 4.1: DOCX Export
- `renderMeetingDocx(meeting)` utility producing a .docx buffer
- Same section structure as Markdown: title, overview, decisions, action items, transcript, metadata
- Export button in meeting detail alongside existing MD/PDF
- API endpoint: `GET /api/meetings/:id/export/docx`

#### Feature 4.2: Public Share Page Polish
- Branded header with MeetSum logo and meeting title
- Section toggles respect `includedSections` from share settings
- Quality warnings shown when applicable
- Mobile-responsive layout
- Provider metadata footer
- "Powered by MeetSum" footer link

#### Feature 4.3: Notion Export (API Push)
- `POST /api/meetings/:id/export/notion` with Notion database ID
- Creates a Notion page with meeting sections as blocks
- Requires user to configure Notion integration token in settings
- Status tracking in export history

---

### Epic 5: Operational Hardening (P1 — Medium Value)

#### Feature 5.1: Unified Backup/Restore
- `npm run backup:full` command that:
  - Dumps Postgres to timestamped SQL file
  - Snapshots MinIO buckets to local tar
  - Writes a manifest JSON with backup metadata
- `npm run backup:restore --from <path>` to restore both
- Documented in `docs/backup-restore.md` (update existing)

#### Feature 5.2: Media Retention Enforcement
- Scheduled worker job (`media.retention.enforce`) running daily
- Default policy: audio retained 180 days, video retained only if explicitly opted-in
- Transcripts and summaries retained indefinitely
- Dry-run mode with `--dry-run` flag
- Audit log entry for each deletion

#### Feature 5.3: GitHub Actions Auto-Deploy
- Workflow triggered on `main` push
- SSH into VPS, `git pull`, `docker compose up -d --build app worker`
- Gated by `DEPLOY_ENABLED` secret (default: disabled)
- Slack/webhook notification on deploy completion

## Non-Functional Requirements

### Performance
- Room detail page loads in < 500ms for rooms with ≤ 50 meetings
- Rate limiter adds < 1ms overhead per request
- Backup command completes in < 5 minutes for ≤ 10GB total data

### Security
- API keys stored as bcrypt hashes, never logged or exposed after creation
- Rate limiting prevents brute-force on public share tokens
- Audit log covers all security-relevant actions
- DOCX export strips media URLs (same as MD/PDF)

### Scalability
- Rate limiter designed with Redis upgrade path (currently in-memory)
- Backup/restore works with any S3-compatible storage, not just MinIO

### Accessibility
- Room detail page meets WCAG 2.1 AA contrast ratios
- Speaker assignment modal keyboard-navigable
- Public share page uses semantic HTML with proper heading hierarchy

## Technical Constraints

- Must maintain backward compatibility with v0.2.0 database schema
- All new migrations must be additive (no column drops or renames)
- n8n integration requires n8n ≥ 1.97.0 (MCP support)
- Notion export requires user to set up a Notion integration independently
- Desktop recorder is Electron-only; no web-based recording
- VPS deployment remains single-instance (no multi-node)

## Success Metrics

- **Admin self-service:** API key create/revoke works from UI without SSH
- **Rate limiting live:** Public share returns 429 after 60 rapid requests
- **Template wiring:** A "sales" meeting generates sales-specific intelligence
- **Room usability:** At least 3 production rooms with ≥ 2 meetings each
- **Backup reliability:** Full backup/restore cycle tested on VPS
- **Test coverage:** ≥ 200 tests passing (currently 180)

## Open Questions

1. **Notion export priority:** Is Notion export worth the complexity of managing a third-party integration token, or should we defer to v0.4.0?
2. **Desktop recorder:** Should this be a separate repo or a monorepo package?
3. **Vertex AI:** Is the cost differential significant enough to justify the migration effort now, or wait until Google AI Studio pricing changes?
4. **Auto-deploy:** Should auto-deploy be enabled by default on main push, or always require manual approval via a GH Actions workflow_dispatch?
