# Story: API Key Management UI

## Epic: Admin Self-Service
## Priority: P0
## Status: todo

## Description

Build a settings page at `/settings/api-keys` that allows admins to create, list, and revoke API keys without SSH access. This closes the last admin-facing gap — currently API keys can only be managed via environment variables or database commands.

## Acceptance Criteria

- [ ] Settings page shows a table of active API keys with: label, prefix (first 8 chars), created date, last used date, expiration
- [ ] "Create Key" button opens a dialog with label + optional expiration inputs
- [ ] On creation, the raw key is shown once with a copy-to-clipboard button, then never again
- [ ] "Revoke" button marks the key as revoked with a confirmation dialog
- [ ] Revoked keys no longer authenticate API requests
- [ ] Each create/revoke action records an audit log entry
- [ ] Test: create → authenticate → revoke → reject flow

## Technical Notes

- Existing `lib/auth/api-keys.ts` has hashing logic — extend with CRUD operations
- Store keys in `api_keys` table (new migration)
- Use bcrypt for key hashing (already a project dependency pattern)
- API routes: `GET/POST /api/admin/api-keys`, `DELETE /api/admin/api-keys/:id`
- UI: extend the existing Settings page layout

## Dependencies

- None (self-contained)

## Files Affected

- `db/migrations/XXX_create_api_keys_table.sql` — new migration
- `lib/auth/api-keys.ts` — extend with CRUD
- `app/api/admin/api-keys/route.ts` — new: list + create
- `app/api/admin/api-keys/[id]/route.ts` — new: revoke
- `app/(app)/settings/api-keys/page.tsx` — new: UI page

# Story: Rate Limit Middleware Integration

## Epic: Admin Self-Service
## Priority: P0
## Status: todo

## Description

Wire the existing `lib/rate-limit.ts` sliding window rate limiter into the Next.js middleware so that rate limiting is automatically enforced on public and resource-intensive endpoints. Currently the rate limiter exists as a tested library but is not applied to any routes.

## Acceptance Criteria

- [ ] Public share pages (`/share/:token`) are limited to 60 req/min per token
- [ ] AI processing endpoints are limited to 10 req/min per IP
- [ ] Export endpoints are limited to 20 req/min per IP
- [ ] Ask/memory endpoints are limited to 30 req/min per IP
- [ ] Blocked requests return `429 Too Many Requests` with `Retry-After` header
- [ ] All API responses include `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset` headers
- [ ] Rate limiting can be disabled via `MEETSUM_RATE_LIMIT=false` env var
- [ ] Test: verify 429 response after exceeding limit

## Technical Notes

- Modify `middleware.ts` to call `checkRateLimit()` before route handling
- Use IP address as rate limit key for authenticated routes
- Use share token as rate limit key for public share pages
- Apply `applyRateLimitHeaders()` to responses
- Presets already defined in `RATE_LIMIT_PRESETS`

## Dependencies

- None (lib/rate-limit.ts already complete and tested)

## Files Affected

- `middleware.ts` — add rate limit checks
- `lib/rate-limit.ts` — add IP extraction helper
- `tests/rate-limit-middleware.test.ts` — new: middleware integration tests

# Story: Unified Backup/Restore

## Epic: Operational Hardening
## Priority: P1
## Status: todo

## Description

Create a single backup command that captures both Postgres and MinIO data, and a corresponding restore command. Currently operators must manually run separate commands for each service.

## Acceptance Criteria

- [ ] `npm run backup:full` creates a timestamped backup directory with: Postgres SQL dump, MinIO bucket snapshot, manifest.json
- [ ] `npm run backup:restore --from <path>` restores both Postgres and MinIO from a backup directory
- [ ] Manifest includes: timestamp, postgres version, meetsum version, file sizes, checksums
- [ ] Backup completes in < 5 minutes for ≤ 10GB data
- [ ] Restore validates manifest before applying
- [ ] Documented in updated `docs/backup-restore.md`
- [ ] Test: backup → corrupt data → restore → verify data integrity

## Technical Notes

- Use `pg_dump` via child_process for Postgres
- Use `mc mirror` for MinIO bucket snapshot
- Store backups in `/opt/meetsum/backups/` on VPS
- Manifest format: `{ timestamp, version, files: [{name, sizeBytes, checksum}] }`

## Dependencies

- `mc` (MinIO client) must be available in the container

## Files Affected

- `scripts/backup.ts` — new: backup script
- `scripts/restore.ts` — new: restore script
- `docs/backup-restore.md` — update with new commands
- `package.json` — add `backup:full` and `backup:restore` scripts
