# MeetSum V0.1.0 Development Plan

Date: May 18, 2026.

## Goal

Finish MeetSum v0.1.0 as a daily-usable single-admin meeting intelligence app: selected Google Meet/Drive/manual content becomes reliable Hebrew/mixed transcript intelligence, can be reviewed and shared, and can flow into RealizeOS and webhook/n8n automation.

## Phase 1: Debug Local Whisper Fallback

Priority: highest.

Problem: the latest real Hebrew/mixed recording attempted `local-whisper` and fell back to Gemini. The fallback is visible, but the root cause is unknown.

Implementation tasks:

- Reproduce with one selected real recording only.
- Inspect worker logs for `audio.transcribe` around the failed local ASR attempt.
- Inspect faster-whisper logs for model load, request start, request completion, restart, timeout, or memory failure.
- Verify `LOCAL_TRANSCRIPTION_TIMEOUT_MS=900000` is present in production.
- Check whether the worker receives a malformed response, an HTTP error, a timeout, or a connection drop.
- Add or update tests so fallback metadata includes attempted provider, actual provider, fallback flag, and error reason.
- Update documentation with the root cause and current operational recommendation.

Acceptance criteria:

- The failure reason is precise enough to act on.
- A Hebrew/mixed sample either completes through local Whisper or falls back with a recorded reason.
- Gemini fallback remains active.
- No raw recording or transcript is committed.

Verification:

```bash
npm test
npm run typecheck
npm run lint
npm run build
```

## Phase 2: Provider Metadata In Share And Exports

Priority: high.

Problem: provider metadata and quality warnings exist in meeting payloads and RealizeOS payloads, but exported/shared outputs need clearer trust context.

Implementation tasks:

- Add a "Processing and quality" section to Markdown export.
- Add a concise provider/quality block to PDF export.
- Add a safe public-share technical details section if it does not clutter the page.
- Include provider, attempted provider, model, fallback used, confidence, latency when available, and quality warnings.
- Keep audio/video private on public share pages.

Acceptance criteria:

- Markdown export includes provider metadata and warnings.
- PDF export includes provider metadata and warnings.
- Public share page remains readable and exposes no media URLs.

Verification:

```bash
npm test
npm run typecheck
npm run lint
npm run build
```

Smoke:

- Open one processed public share page unauthenticated.
- Render Markdown and PDF for the same meeting.
- Confirm no private media URL is visible.

## Phase 3: Meet Artifact Discovery, Import, And Recovery UX

Priority: high.

Problem: Google Meet artifacts are the intended v0.1.0 live capture path, but users need clearer freshness, next actions, and recovery.

Implementation tasks:

- Show Meet artifact sync freshness in Workspace.
- Show counts for conference records, recordings, transcripts, smart notes, linked artifacts, and importable artifacts.
- Improve meeting empty/upcoming states:
  - no artifact found,
  - artifact available,
  - processable smart notes,
  - processable recording,
  - manual upload needed.
- Add grouped failed-job recovery by meeting, stage, and error.
- Ensure stale failed jobs do not make later completed meetings look failed.
- Keep automatic media download disabled.

Acceptance criteria:

- A meeting with artifacts has a clear Process from Google artifacts action.
- A meeting without artifacts has an actionable checklist.
- Failed jobs are retryable from Workspace.
- Completed meetings remain completed even when older jobs failed.

Verification:

```bash
npm test
npm run typecheck
npm run lint
npm run build
```

Production smoke:

- Run Meet artifact sync.
- Select one artifact.
- Import/process only that selected artifact.
- Confirm status and recovery UI.

## Phase 4: RealizeOS And Webhook/N8n Polish

Priority: high because automations are v0.1.0 scope.

Problem: backend surfaces exist, but the product needs clearer preview, send, retry, and delivery history.

Implementation tasks:

- Add RealizeOS payload preview for selected meeting.
- Show export status: queued, sent, failed.
- Show response body or error body when safe.
- Add retry from meeting integration surface and Automations page.
- Improve webhook subscription UI: create, list, pause/resume, send test, view delivery history, retry failed delivery.
- Add copyable n8n setup guidance.
- Keep external-impact actions approval-gated.

Acceptance criteria:

- A processed meeting can be previewed and queued to RealizeOS.
- RealizeOS export history is visible and retryable.
- A webhook test can be sent to an approved endpoint.
- Delivery history records status and errors.

Verification:

```bash
npm test
npm run typecheck
npm run lint
npm run build
```

Smoke:

- Preview RealizeOS payload for one processed meeting.
- Queue export only when approved.
- Send webhook test to approved n8n/test URL.

## Phase 5: Memory, Rooms, And Hebrew Quality Loop

Priority: medium-high.

Problem: Memory and Rooms exist, but they need to feel useful for daily work.

Implementation tasks:

- Add Memory filters by room, participant, tag, language, status, and date.
- Improve Ask all meetings ranking and citations.
- Add Room detail actions: linked meetings, open tasks, recurring participants, related artifacts, and room-specific Ask.
- Add room suggestions from title, recurring event, tags, participants, and Drive/Meet context.
- Create private ASR evaluation manifest template documentation only; do not commit samples.
- Tighten task extraction tests for "no transcript fragments as tasks."

Acceptance criteria:

- Memory answer citations open the source meeting.
- At least one Room can function as a real workstream hub.
- Private Hebrew/mixed evaluation can be run repeatedly.

Verification:

```bash
npm test
npm run typecheck
npm run lint
npm run build
npm run asr:evaluate -- --manifest .secrets/asr-eval/manifest.json
```

Only run `asr:evaluate` when private samples exist locally.

## Release Gate

Before declaring v0.1.0:

- One real selected Google Meet artifact/import/process/share smoke passes.
- One manual upload or Drive import smoke passes.
- Hebrew/mixed provider behavior is understood and documented.
- Public share excludes private media.
- Markdown/PDF exports include core meeting sections and quality/provider context.
- RealizeOS export is previewable and auditable.
- Webhook test delivery works against an approved endpoint.
- Memory search/Ask returns citations.
- Rooms can link real meetings and tasks.

Required commands:

```bash
npm test
npm run typecheck
npm run lint
npm run build
npm audit --omit=dev
```

Production deployment requires:

```bash
cd /opt/meetsum
./scripts/backup-postgres.sh
./scripts/deploy-vps.sh
curl -fsS https://meetsum.realization.co.il/api/health
```

Do not deploy without a Postgres backup.
